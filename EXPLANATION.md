# Task Manager — *Система управления задачами*
Документ описывает внутреннее устройство проекта **Task Manager**, логику работы основных функций и ключевые технические решения.
______
## :open_file_folder: *Структура проекта*
Проект построен на Qt 5.15.3 и разделён на логические модули:
+ qmake-проект
+ Заголовочные файлы (.h)
+ Исходники (.cpp)
+ Файл-ресурсов
+ README.md
+ EXPLANATION.md
+ .gitignore

Описание каждого модуля:
+ **TASK_manager.pro** — файл конфигурации qmake, где указаны модули (widgets, core, gui), стандарт C++11, пути к исходникам и настройки компилятора GCC 4.2.
+ **main.cpp** — точка входа в приложение, инициализирует QApplication и отображает главное окно.
+ **task.h** — каждая задача представляется в вде структуры.
+ **mainwindow.h / mainwindow.cpp>** — основной класс интерфейса. Он связывает визуальные компоненты с логикой обработки данных и файловой системой.
+ **taskList.h / taskList.cpp** — компонент визуализации и управления списком.
Этот класс является ключевым пользовательским виджетом, который отвечает за отображение задач, их хранение в памяти и обработку действий пользователя (редактирование, удаление, сортировка, фильтрация).
+ **dialogCreate.h / dialogCreate.cpp** — диалог создания задачи.
Этот класс представляет собой модальное окно, предназначенное для сбора данных от пользователя при создании новой задачи и передать ее в основной список.
+ **dialogEdit.h / dialogEdit.cpp** — диалог редактирования задачи.
Этот класс отвечает за изменение уже существующих задач. Он выглядит почти так же, как окно создания, но работает по принципу «загрузить данные — изменить — вернуть результат».
+ **dialogOpen.h / dialogOpen.cpp** — окно выбора и открытия файла
Этот класс представляет собой диалоговое окно, которое позволяет пользователю выбрать существующий JSON-файл с задачами на компьютере и загрузить его в приложение.
+ **icons.qrc** — это Qt Resource Collection, специальный ресурсный файл, который используется для хранения и подключения внутренних ресурсов проекта(иконок)

## :paperclip: *Как реализовано добавление задачи*
Процесс добавление задачи проходит через несколько уровней взаимодействия:

+ **Вызов:** Пользователь нажимает кнопку «Создать» в MainWindow, что открывает модальное окно DialogCreate.
+ **Ввод:** Пользователь заполняет поля (название, описание, дедлайн, статус).
+ **Валидация:** При нажатии «Создать» внутри диалога проверяется поле названия. Если оно пустое, выводится предупреждение.
+ **Передача:** Если всё верно, диалог испускает сигнал taskCreated(const Task &task), передавая заполненный объект Task.
+ **Прием:** MainWindow получает этот сигнал и перенаправляет задачу в taskList->addTask(task).
+ **Отображение:** Класс TaskList присваивает задаче уникальный ID, добавляет её в свой внутренний вектор и создает элемент QTreeWidgetItem для отображения в списке.

## :inbox_tray: *Как работает сохранение и загрузка данных*
Приложение использует формат JSON, что обеспечивает читаемость данных и легкость их обработки.
+ ***Сохранение*** TaskList → JSON-массив → JSON-объект → файл
	+ 1. MainWindow вызывает метод myTaskList->JSArr().
	+ 2. Программа проходит циклом по всем задачам в QVector<Task>, превращая каждую задачу в QJsonObject (через функцию taskToJS).
	+ 3. Все объекты собираются в массив QJsonArray, который записывается в QJsonDocument.
	+ 4. С помощью класса QFile данные записываются в файл с расширением .json. Если файл не удается открыть, выводится предупреждение через QMessageBox.
	
+ ***Загрузка***	Файл → JSON → массив → TaskList → обновление UI
	+ 1. Текст из файла преобразуется обратно в QJsonDocument. На этом этапе работает встроенная проверка на ошибки (QJsonParseError) — если JSON поврежден, пользователь увидит сообщение об ошибке.
	+ 2. Программа проверяет, содержит ли файл объект с массивом "tasks". Это защищает приложение от вылета при попытке прочитать неподходящий файл.
	+ 3. Массив передается в loadFromJSArr(), где для каждого элемента вызывается функция taskFromJS.
	+ 4. При загрузке программа автоматически пересчитывает nextId. Это гарантирует, что новые задачи получат уникальные идентификаторы, которые не будут конфликтовать со старыми.
	+ 5. После заполнения вектора allTasks вызывается метод refreshList(), который заново отрисовывает все задачи в списке.

## :round_pushpin: *Что такое сигналы и слоты? Как они используются в проекте?*
**Сигналы и слоты** — это механизм Qt для связи между объектами.


### ***Использование в проекте Task Manager:***
В данном приложении сигналы и слоты используются для связи интерфейса с логикой данных.
1. **Связь окон:** Передача данных между окнами:
+ при добавлении задачи (DialogCreate->TaskList)
+ при редактировании задачи (TaskList->DialogEdit)
2. **Обработка кнопок интерфейса:** В конструкторе MainWindow кнопки соединяются с логикой приложения
3. **Динамические действия в списке** Для кнопок «Удалить» и «Редактировать», которые создаются динамически для каждой задачи, используется передача ID
4. **Работа с файлами:** В классе DialogOpen реализован сигнал fileOpened(QString). Как только файл успешно прочитан с диска, этот сигнал передает весь считанный текст напрямую в слот displayFileContent главного окна для дальнейшего парсинга в JSON.

## :interrobang: *С какими трудностями вы столкнулись?*
1. **Сложная структура списка в QTreeWidget:**

	***Проблема:*** Нужно было отобразить не просто текст, а интерактивные кнопки («Удалить», «Редактировать») и раскрывающееся описание внутри одной строки.

	***Решение:*** Использование метода setItemWidget для вставки кнопок в ячейки дерева и создание иерархической связи «родитель-потомок» для реализации скрытого описания задачи.
	
2. **Идентификация динамических кнопок:**

	***Проблема:*** Когда в списке 10 кнопок «Удалить», программа должна точно знать, какая из них нажата и к какой задаче она относится.

	***Решение:*** Использование механизма setProperty("taskId", id) для кнопок. Это позволило в одном слоте через sender() получать ID задачи и находить её в векторе allTasks.

3. **Сериализация вложенных данных в JSON:**

	***Проблема:*** JSON не умеет напрямую хранить типы данных Qt (например, QDateTime или enum).

	***Решение:*** Написание функций-преобразователей taskToJS и taskFromJS, которые переводят дату в формат ISO (Qt::ISODate), а статусы задач — в строковый вид и обратно.
	
4. **Управление памятью диалоговых окон:**

	***Проблема:*** При частом открытии календаря или окон создания задач могли возникнуть утечки памяти.

	***Решение:*** Использование атрибута Qt::WA_DeleteOnClose и проверка указателей (например, calendarDialog = nullptr), чтобы приложение корректно очищало ресурсы после закрытия окон.
